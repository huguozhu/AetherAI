渲染引擎架构方案对比

基于：C++20 Modules / Windows(D3D12→Vulkan) / 研究学习项目 / CMake

方案 A：分层库架构 推荐

Aether.Core
Aether.RHI
Aether.Resources
Aether.Renderer
Aether.Shaders

多个 CMake STATIC 库，每个库内部使用 C++20 Modules。严格分层依赖：Core → RHI → Resources → Renderer → Shaders。

优势

  • 构建隔离好，修改底层只重编相关库
  • 清晰的依赖方向，防止循环依赖
  • 每一层可独立测试和理解
  • Modules 粒度适中，避免巨型 Module

代价

  • CMake 跨 target 传递 Module 依赖需要额外配置
  • 模块间循环引用需要接口层隔离

方案 B：单体模块项目

单一 CMake 目标，所有源代码按目录结构组织，通过 C++20 Modules 分区（Partition）来划分内部模块。

模块分区结构
Aether          // 主模块接口
├── Aether:Core          // 基础类型、数学、内存
├── Aether:RHI           // 命令列表、GPU 资源抽象
├── Aether:RHI:D3D12     // D3D12 实现分区
├── Aether:RHI:Vulkan    // Vulkan 实现分区
├── Aether:Resources     // 异步资源加载
├── Aether:Renderer      // GPU Driven Pipeline
└── Aether:Shaders       // HLSL 编译与反射

优势

  • Modules 分区间共享类型更直接
  • 构建配置简单，单个 CMake target
  • 适合学习——所有代码在一个命名空间视野内

代价

  • 修改任何部分都需要重编整个项目（随规模变大痛点明显）
  • 编译器对 Module Partitions 的支持仍在演进
  • 无法利用并行构建隔离
A

分层库架构 推荐

多个 CMake 库，严格分层依赖，构建隔离好

B

单体模块项目

单一 CMake 目标，通过 Modules Partitions 组织

我的推荐：方案 A（分层库架构）。原因：(1) C++20 Modules 目前跨 target 的支持已趋于成熟（CMake 3.28+ / VS 17.12+ / Clang 18+）；(2) 学习项目也需要好的构建体验，避免后期每次修改等待全量重编；(3) 未来如果要给不同项目复用渲染层，分层架构天然支持。
